## CHANGE 1 OF 2: Replace fetchEventRecommendation function
=========================================================================

### FROM (current bare code in APIServer2.5.js and api-component2.0.js):

```javascript
// Fetch full recommendation for a single event by ID
async function fetchEventRecommendation(eventId) {
    const options = {
        hostname: API_BASE_URL,
        path: `/${API_VERSION}/irrigation-events/${eventId}.json`,
        method: 'GET',
        headers: { 'Authorization': `Bearer ${apiToken}` }
    };
    try {
        const response = await makeRequest(options);
        if (response.statusCode === 200) return response.data;
    } catch (error) {
        console.error(`Error fetching event ${eventId}:`, error.message);
    }
    return null;
}
```

### TO (calls through MCP tool registry):

```javascript
// Fetch full recommendation for a single event by ID
// Calls through MCP tool registry — no bare makeRequest
async function fetchEventRecommendation(eventId) {
    if (!apiToken || !eventId) return null;
    const tool = server.tools.get('get_event_recommendation');
    if (!tool) return null;
    const result = await tool.handler({ eventId: String(eventId) });
    return result.success ? result.event : null;
}
```

### WHY
- FROM: bare makeRequest with hardcoded API_BASE_URL — same pattern as client bare fetch
- TO: calls through registered MCP tool — consistent with how every other call should work
- The CropManage API script (hostname, path, method, headers) moves INTO the tool handler
- fetchEventRecommendation becomes a thin wrapper that calls the tool

## CHANGE 2 OF 2: Add get_event_recommendation to apiServiceTools
=========================================================================

### FROM (current — tool does not exist):
batch_post_queue is the last tool in the array. No get_event_recommendation.

### TO (add after batch_post_queue, before closing ]):

```javascript
    {
        name: "get_event_recommendation",
        description: "Fetch full recommendation for a single irrigation event by event ID. Returns enriched event data with RecommendationSummary.",
        inputSchema: {
            type: "object",
            properties: {
                eventId: {
                    type: "string",
                    description: "CropManage irrigation event ID"
                }
            },
            required: ["eventId"]
        },
        handler: async (args) => {
            if (!apiToken) return { success: false, message: 'No token available' };
            if (!args.eventId) return { success: false, message: 'eventId required' };
            const options = {
                hostname: API_BASE_URL,
                path: `/${API_VERSION}/irrigation-events/${args.eventId}.json`,
                method: 'GET',
                headers: { 'Authorization': `Bearer ${apiToken}` }
            };
            try {
                const response = await makeRequest(options);
                if (response.statusCode === 200) {
                    return { success: true, event: response.data };
                }
                return { success: false, message: `Event fetch failed: ${response.statusCode}` };
            } catch (error) {
                return { success: false, message: error.message };
            }
        }
    }
```

### WHY
- CropManage API script (hostname, path, method, headers) lives inside the tool handler
- Same metadata format as all other tools: name, description, inputSchema, handler
- makeRequest stays in ONE place — inside the registered tool
- Tool is callable by name through server.tools.get() — MCP pattern
- Exposed at /tools/get_event_recommendation endpoint automatically
